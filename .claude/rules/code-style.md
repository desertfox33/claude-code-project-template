# Code Style Rule

Use this rule when editing source code.

## Standards

- Prefer clear, maintainable code over compact cleverness.
- Keep functions focused on one responsibility.
- Use names that explain intent.
- Avoid broad refactors unless the task asks for them.
- Preserve existing project conventions unless they are unsafe or broken.
- Add comments only when they explain a decision, constraint, or non-obvious behavior.

## Output expectation

When code changes are complete, summarize:
- What changed
- Why it changed
- Files touched
- Tests or checks run
- Known risks or follow-up work
